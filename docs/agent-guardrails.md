# Agent Guardrails

## Goal

Ship portfolio updates that feel specific, credible, and evidence-backed.

## Non-Negotiables

- Do not publish draft-status language on the site.
- Do not claim outcomes that are not supported by local source material.
- Do not invent AI-tool usage details, shipped features, or ownership scope.
- Do not frame Hardik as a visual designer if the source material supports a product/operator story instead.
- Do not use generic filler phrases like:
  - "turn ambiguity into clarity"
  - "bridge strategy and execution"
  - "artifact people can react to"
  unless followed immediately by concrete evidence.

## Preferred Positioning

Hardik is most credible as:

- a product-minded operator
- a technical generalist with strong written thinking
- someone who can structure ambiguous work and move it toward a testable artifact

Hardik is less credible as:

- a polished visual designer
- an elite full-stack engineer first
- a pure strategist with no implementation depth

## Proof Standard

Every major claim on the site should be backed by at least one of:

- resume or CV entry
- local repo README
- mockup, deck, or written business-plan artifact
- internal critique or tailored application material that independently confirms the same story

## Feature Rule

Standout features must be useful, not decorative.

Examples:

- good: a command surface that lets a reviewer inspect projects, proof, and links
- bad: a fake terminal that types generic slogans

## Writing Rule

Prefer:

- concrete nouns
- constrained claims
- measurable results
- visible tradeoffs

Avoid:

- inflated comparisons
- overclaiming ownership
- high-density abstract vocabulary
